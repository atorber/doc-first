# 5. 代码审查指南

## 5.1 审查流程

1. **创建 PR**: 开发者完成功能后创建 Pull Request
2. **自动检查**: CI 自动运行测试和 Lint
3. **Code Review**: 至少一名团队成员审查
4. **合并**: 审查通过后合并到主分支

## 5.2 审查要点

### 功能性
- [ ] 功能按需求实现
- [ ] 边界条件处理正确
- [ ] 错误处理完善

### 代码质量
- [ ] 符合编码规范
- [ ] 无重复代码
- [ ] 变量命名清晰
- [ ] 适当的注释

### 测试
- [ ] 有单元测试
- [ ] 有集成测试（如稀缺）
- [ ] 所有测试通过
- [ ] 测试覆盖率达标

### 性能
- [ ] 无明显的性能问题
- [ ] 数据库查询优化

### 安全性
- [ ] 无安全漏洞
- [ ] 敏感信息未泄露

## 5.3 审查意见处理

### 必须修改
- 功能性缺陷
- 安全漏洞
- 严重的性能问题

### 建议修改
- 代码风格问题
- 命名优化
- 添加注释

### 可选优化
- 性能微调
- 代码重构
